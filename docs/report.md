## 1. 배경

### 1.1. 분석 배경 및 목적

로스트아크는 직업별 밸런스 이슈가 지속적으로 커뮤니티 내 핵심 논제로 등장하는 게임입니다. 패치마다 특정 직업이 버프 또는 너프되고, 이에 대한 유저들의 반응은 각 직업 게시판에 대규모로 축적됩니다. 이러한 여론을 정량화할 수 있는 모델을 개발하고자 하였습니다.

이 프로젝트는 직업별 커뮤니티 게시글을 수집하고, 각 게시글에서 밸런스 관련 표현을 추출하여 긍정/부정/중립으로 분류하는 한국어 감성 분석 모델을 구축합니다. 이를 통해 각 직업에 대한 유저 여론을 수치화하고, 직업 간 밸런스 만족도를 비교 분석하는 것이 목적입니다.

### 1.2. 수집 대상 및 분석 단위

- **수집 대상:** 인벤 로스트아크 직업별 게시판 29개, 게시글 제목/본문/댓글
- **분석 단위:** 게시글 단위가 아닌 **expression 단위** — 하나의 게시글에서 밸런스 관련 표현을 N개 추출하여 각각 독립적으로 분류합니다. 이는 한 게시글 안에서도 긍정의 표현과 부정의 표현이 섞여 등장하는 커뮤니티의 특성을 반영합니다.

### 1.3. 사전 도메인 지식

본 분석의 이해를 위해 주요 용어를 정의합니다.

- **버프/너프:** 게임사의 직업 강화/약화 패치. 커뮤니티에서 버프 요청 또는 너프 불만이 감성 분류의 핵심 대상입니다.
- **서포터 직업:** 바드, 홀리나이트, 도화가, 발키리. 딜이 아닌 파티 지원이 주 역할이며, 밸런스 논의 패턴이 딜러와 상이합니다.
- **딜러 직업:** 서포터 4종을 제외한 나머지 직업. 딜량, 유틸리티 등이 밸런스 논의의 주요 대상입니다.
- **드러눕기 시위 (`●▅▇█▇▆▅▄▇`):** 유저가 강한 불만을 표현할 때 사용하는 특수 이모지. 본 모델은 이를 negative 표현으로 분류합니다.
- **직업 슬랭:** 붕쯔, 분망 등 커뮤니티 고유 약어. 라벨링 프롬프트에 반영하였습니다.

## 3. 데이터 파이프라인

### 3.1. 크롤링

인벤 로스트아크 직업별 게시판을 Selenium + BeautifulSoup 기반 크롤러로 수집하였습니다. 수집 항목은 게시글 ID, 직업명, 제목, 본문, 댓글, 날짜이며, 직업당 최신 게시글 순으로 수집하였습니다.

| 항목 | 값 |
| --- | --- |
| 수집 일시 | 2026.03.20 |
| 수집 게시판 수 | 29개 직업 |
| 수집 게시글 수 | 12,661건 |
| unique post_id | 2,929건 |

수집된 원본 CSV(`lostark_crawled_20260320_1305.csv`)는 12,661행이지만, 동일 게시글이 다수의 댓글 행으로 수집되는 구조로 인해 unique post_id는 2,929건입니다.

### 3.2. 라벨링 파이프라인

라벨링은 3단계로 구성됩니다. Groq API가 expression 추출과 초안 분류를 담당하고, 신뢰도가 낮은 항목은 규칙 기반 후처리로 재분류하며, 최종적으로 판단이 어려운 케이스만 수동 검수합니다.

```
1단계: Groq LLM       → 게시글에서 밸런스 관련 표현 추출 + 초안 분류
2단계: auto_review.py → 신뢰도 < 0.75 항목에 키워드/패턴 기반 재분류
3단계: label_reviewer → 1·2단계에서 판단 불가한 소수 케이스 수동 검수
```

**단계별 설계 의도**

- **1단계 (Groq):** LLM은 expression 추출에 강점이 있습니다. 한국어 비정형 게시글에서 밸런스 관련 표현만 선별하는 작업은 패턴만으로 대체하기 어렵습니다. 분류 결과는 신뢰도와 함께 저장합니다.
- **2단계 (auto_review):** Groq 무료 티어 한도로 인해 신뢰도가 낮은 항목이 다수 발생합니다. 이를 분석한 결과, 낮은 신뢰도 항목은 대부분 두 가지 패턴으로 구분됩니다. 각 패턴은 아래에 서술 된 규칙에 따라 라벨링을 진행하였습니다.
    
    ① 버프/너프 키워드가 명확한 경우(regex로 분류 가능)
    
    ② 밸런스 키워드 없이 빌드/수치/잡담 위주인 경우(unrelated로 처리)
    
- **3단계 (수동):** 위 두 단계 이후에도 판단이 어려운 케이스만 수동 처리합니다. 전체 NEEDS_REVIEW 1,273건 중 2단계에서 1,261건(99.1%)을 자동 처리하고 12건만 수동 검수하였습니다.

**Groq 무료 티어 제약 및 운영 전략**

| 항목 | 값 |
| --- | --- |
| API 키 수 | 3개 (멀티 계정) |
| 키당 일일 한도 | 1,000 RPD (Requests Per Day) |
| 합산 일일 처리량 | 최대 3,000건/일 |
| 리셋 시각 | UTC 00:00 (KST 09:00) |

라벨링이 3,000건/일을 초과할 경우 자동으로 저장 후 종료하며, 다음 실행 시 처리 완료 post_id를 건너뛰고 이어서 처리합니다.

## 4. 모델 설계

### 4.1. 사전학습 모델 선정

**`monologg/koelectra-base-v3-discriminator`**

ELECTRA 아키텍처는 모든 토큰에 학습 신호를 생성하는 Generator-Discriminator 구조로, MLM 기반 BERT 대비 데이터 효율이 높습니다. 한국어 NLP 벤치마크(NSMC, KorSTS)에서 KoBERT 동등 이상의 성능을 보이며, expression 단위 단문 분류에 적합한 특성을 가집니다.

### 4.2. 입력 형식

직업 컨텍스트를 입력에 포함하여 직업별 슬랭 및 표현 맥락을 반영합니다.

```
직업: {job_class} 표현: {expression_text}
예시: "직업: 디스트로이어 표현: 디트 너무 약해졌다 버프해줘"
```

### 4.3. 모델 구성

| 모델 | 학습 데이터 | 목적 |
| --- | --- | --- |
| 통합 (`unified`) | 전체 직업 | 베이스라인, 전체 만족도 집계 |
| 딜러 (`dealer`) | 딜러 직업 전체 | 딜러 특화 감성 분류 |
| 서폿 (`support`) | 바드 / 홀리나이트 / 도화가 / 발키리 | 서포터 특화 감성 분류 |
| 직업 개별 | 데이터 상위 직업 3~4개 | 단일 직업 특화, 통합 모델과 성능 비교 |

직업 개별 모델 대상은 라벨링 완료 후 직업별 데이터 분포 확인 후 선정합니다.

### 4.4. 주요 하이퍼파라미터

| 파라미터 | 값 | 근거 |
| --- | --- | --- |
| MAX_LENGTH | 128 | expression 단위 입력 30~60자 충분 커버 |
| TRAIN_BATCH | 16 | RTX 4060 8GB VRAM 안정 한계 |
| LR | 3e-5 | BERT 계열 권장 범위(1e-5~5e-5) 중간값 |
| WARMUP_RATIO | 0.1 | 초기 파국적 망각 방지, 원 논문 권장값 |
| EPOCHS | 5 | Early Stopping(PATIENCE=2)과 병행 사용 |
| Class Weight | Balanced | 클래스 불균형 대응, Macro F1 최적화 |
